# Plan de auditoría y corrección de generación de códigos de acceso

## 1. Auditoría del código existente
- **Buscar** la implementación del servicio `ICodigoAccesoService` (métodos `generar`, `validar`, `hashear`, `getLongitudPorRol`).
- **Revisar** los lugares donde se llama a `codigoService.generar(rol)` (creación de usuarios y botón de generación en la UI).
- **Verificar** que la longitud del código generado depende del rol:
  - `admin` y `coordinador` → 6 dígitos.
  - `tecnico` → 5 dígitos.
- **Comprobar** que la validación (`validar`) usa la longitud adecuada.
- **Confirmar** que el hash se guarda y que al devolver el usuario se incluye el código en texto plano solo cuando corresponde.

## 2. Corrección de la generación de códigos
- Implementar/actualizar `generar(rol)` para:
  1. Determinar longitud con `getLongitudPorRol`.
  2. Generar un número aleatorio sin ceros a la izquierda y que sea único (consultar la tabla `usuarios` para colisiones).
  3. Re‑intentar hasta obtener un código no existente.
- Asegurarse de que `validar(codigo, rol)` verifica longitud exacta y que el código es numérico.
- Ajustar la lógica de creación de usuarios (`UsuarioService.crear`) para usar el nuevo generador.

## 3. Actualización del endpoint y UI
- En la ruta que maneja el botón "Generar códigos de acceso" (probablemente `src/routes/*` o controlador asociado), llamar al mismo método de generación y devolver el código recién creado.
- Garantizar que la respuesta contiene `codigo_acceso` y que la tabla de usuarios muestra este valor (sin ocultarlo tras el hash).

## 4. Permisos de coordinador
- Revisar políticas de autorización (middlewares, checks de rol) para:
  - Permitir a **coordinador** editar usuarios con rol `tecnico` y `beneficiario`.
  - Permitir a **coordinador** crear, editar y eliminar actividades, asignaciones y beneficiarios.
- Modificar los controladores/servicios correspondientes (`src/routes/coordinadores.ts`, `src/services/*`) para añadir verificaciones de permiso basadas en `actor.rol`.

## 5. Confirmación de eliminación sin restricciones
- En `UsuarioService.eliminar` y `eliminarFisico` asegurar que la autorización permite la acción salvo que el actor sea el mismo usuario.
- Revisar otros recursos (asignaciones, actividades) para que la eliminación respete la regla de "sin restricción" según el rol del solicitante.
- Implementar una respuesta tipo notificación tipo confirmación (p.ej., mensaje `confirmación: true` en la respuesta) para que el frontend muestre un diálogo de confirmación antes de ejecutar la eliminación.

## 6. Pruebas y validaciones
- Añadir/actualizar pruebas unitarias para:
  - Generación de códigos con longitudes correctas y unicidad.
  - Validación de códigos según rol.
  - Permisos de coordinador sobre técnicos/beneficiarios/actividades/asignaciones.
  - Eliminación de usuarios y recursos con la respuesta de confirmación.
- Ejecutar pruebas y lint para asegurar que no se rompen funcionalidades existentes.

## 7. Documentación
- Actualizar README o documentación interna explicando la lógica de generación de códigos y los permisos de coordinador.
- Añadir comentarios en el código donde se gestionan los códigos de acceso y las validaciones de permisos.
